# OpenClaw Memory Cleanup Rule

Before trimming, shortening, compacting, or restructuring any OpenClaw memory/bootstrap file, preserve the full original content first.

No important project memory should be deleted outright. Detailed memory should be migrated into the OpenClaw Wiki, then memory/bootstrap files should be replaced with concise pointers.

## Required Workflow

1. Save a full timestamped copy of the original memory/bootstrap file.
2. Save a checksum for the original and archived copy.
3. Move detailed project history into the OpenClaw Wiki under Projects, Procedures, Learnings, Runbooks, or Reference.
4. Replace removed sections in memory/bootstrap files with short pointers to the full wiki files.
5. Keep startup memory focused on critical identity, channel bindings, routing rules, security rules, active project pointers, and high-priority operational warnings.
6. Never delete project history unless Justin explicitly confirms it is obsolete.

## Priority Memories to Preserve

- OpenClaw VPS/Mac Mini architecture
- Telegram and WhatsApp account bindings
- ServiceM8 automation work
- Codex Computer Use / Mac Mini desktop control rules
- OpenClaw Wiki / Obsidian / GitHub sync setup
- memory-core and lossless-claw decisions
- Base Lift Services / ServiceM8 service contract project
- agent routing and group access rules
- exec approvals and security posture
- model/provider setup and fallback decisions
